# AGENTS.md

本文件面向后续 AI 编码助手。进入本项目后，先读取并理解 `README.md`、`AIREADME.md` 与本文件；如果文档与实际代码不一致，以实际代码为准，并在合适时更新文档。

## 项目规则

- 生成或修改代码必须使用 UTF-8 编码。
- 编写类或方法时必须写清楚注释，优先使用项目已有的中文 JSDoc/块注释风格。
- 新增变量时，尽量在变量尾部添加简短中文注释说明用途，尤其是状态变量、模块级变量、请求参数和业务常量。
- 编写完成后检查中文内容，确认没有整段中文变成问号乱码，也没有被错误转成 Unicode 转义。
- 优先遵循项目现有风格、架构、组件封装、请求封装和验证方式。
- 提供 URL 链接时，必须先验证链接有效性；无法验证时要明确说明，不要假装可用。
- 如果本文件、`AIREADME.md`、`README.md` 与实际代码不一致，以实际代码为准，并在合适时更新文档。

## 编码与命令

- Windows 终端默认可能按 GBK 读取文本，读取和输出代码文件时必须显式使用 UTF-8，例如 PowerShell 的 `Get-Content -Encoding UTF8`。
- 项目 `.editorconfig` 要求 UTF-8、LF、2 空格缩进、末尾换行。
- 手写代码编辑应保持 UTF-8；不要引入乱码、BOM 争议或无意义的转义。
- 常用命令：

```bash
yarn install
yarn dev
yarn build
yarn lint
yarn preview
```

## 工作方式

1. 先读需求、相关文档和相关代码，不要凭文件名直接猜。
2. 改动前说明假设；存在多种理解时先列出来，必要时询问。
3. 小步修改，直接围绕用户需求落代码。
4. 只改必须修改的文件；不要顺手格式化、重构或清理无关代码。
5. 变更后按风险运行 `yarn lint`、`yarn build` 或更贴近改动的验证命令。
6. 如果发现无关问题，可以在结果中说明，但不要擅自删除或重构。

## 项目架构速记

- `src/main.tsx`：React 入口，挂载 Ant Design 中文配置和 `BrowserRouter`。
- `src/App.tsx`：应用壳，渲染路由，桌面端挂载头部、登录弹窗、信息弹窗、备忘抽屉、英语抽屉。
- `src/routes/index.tsx`：根据 `isMobile()` 切换桌面和移动端路由；当前移动端判断为高大于宽。
- `src/request`：请求封装。登录态业务请求优先使用 `myAxios` 或 `myGet`、`myPost`、`myPut`、`myDelete`。
- `src/store`：MobX Store 和静态默认数据。
- `src/utils`：JWT、本地缓存、日期、URL、图片压缩等工具。
- `src/components`：拖拽、右键菜单、Markdown、favicon 回退、空状态、加载等通用组件。
- `src/pages/Home`：桌面主页、搜索、快捷链接、书签、背景和工具入口。
- `src/pages/MemoDrawer`：桌面备忘录抽屉。
- `src/pages/Mobile`：移动端登录、备忘、博客、个人中心。
- `src/pages/EnglishDrawer`：英语备忘抽屉，复用备忘类型 `4`。
- `src/pages/Blog`：远程 Markdown 博客。
- `src/pages/SeeTime`：观看时间记录和统计图表。
- `src/pages/UtilsPage`：站内工具。

## 业务约定

备忘类型：

| 类型值 | 含义 |
| --- | --- |
| `0` | 普通 |
| `1` | 循环 |
| `2` | 长期 |
| `3` | 紧急 |
| `4` | 备忘英语 |
| `5` | 日记 |
| `6` | 工作 |
| `7` | 其他 |

其他约定：

- 搜索引擎 URL 使用 `@@@` 作为搜索词占位符。
- 英语备忘内容格式为 `英文@@@中文`。
- 循环备忘子项可记录时间、备注和图片列表。
- 用户登录依赖微信公众号验证码，JWT 存在 `localStorage`。
- `localStorageUtils` 会按用户 ID 给缓存键加后缀。
- 桌面端和移动端备忘逻辑并非完全同一套组件，改一端时要检查另一端是否也需要同步。

## 代码风格

- React 函数组件为主，优先复用现有 Ant Design 与 Ant Design Mobile 写法。
- 新的共享状态优先考虑现有 MobX Store；局部 UI 状态用 React Hooks。
- 路径导入优先使用 `@/` 别名。
- TypeScript/TSX 遵循 ESLint：单引号、JSX 属性双引号、React Hooks 推荐规则。
- 后续新增源码文件尽量使用 `ts` 或 `tsx`，除非需要贴合已有 JS/JSX 模块或第三方限制。
- JS/JSX 文件已有较多历史风格，新增代码要贴近当前文件写法。
- 请求失败要通过 `CommonStore.msg`、`Toast` 或当前页面已有提示方式反馈。
- 不要无故改动 `dist`、`node_modules`、锁文件或构建产物。

## 验证清单

- 中文检查：搜索是否出现连续问号乱码、非预期 Unicode 转义。
- 编码检查：新增 Markdown 和源码文件保持 UTF-8。
- 链接检查：新增文档链接前先请求验证，记录无法验证的情况。
- 业务检查：备忘、搜索、书签、登录、移动端入口属于核心路径，相关改动要重点回归。
- 构建检查：涉及 TS/TSX、路由、请求、组件导入时优先运行 `yarn lint` 和 `yarn build`。

## CLAUDE.md 中文翻译追加

来源：[forrestchang/andrej-karpathy-skills/main/CLAUDE.md](https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md)，2026-05-06 验证返回 200。

用于减少常见 LLM 编码错误的行为准则。可按需与项目特定说明合并。

**取舍：**这些准则更偏向谨慎，而不是速度。对于很简单的任务，请自行判断。

### 1. 写代码前先思考

**不要假设。不要隐藏困惑。把取舍摆出来。**

实现之前：

- 明确说明你的假设。不确定时就提问。
- 如果存在多种理解，把它们说出来，不要默默选择其中一种。
- 如果有更简单的方案，就说出来。必要时要提出不同意见。
- 如果有不清楚的地方，停下来。说出困惑点，然后提问。

### 2. 简单优先

**用能解决问题的最少代码。不要写推测性内容。**

- 不实现用户没有要求的功能。
- 不为只使用一次的代码创建抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景编写错误处理。
- 如果你写了 200 行，而 50 行就能解决，请重写。

问自己：“高级工程师会不会认为这过度复杂？”如果答案是会，就简化。

### 3. 手术式修改

**只改必须改的地方。只清理自己造成的问题。**

编辑现有代码时：

- 不要“顺手改进”相邻代码、注释或格式。
- 不要重构没有坏掉的东西。
- 匹配现有风格，即使你自己会用不同写法。
- 如果注意到无关的废弃代码，可以提出来，但不要删除。

当你的改动产生了孤立内容时：

- 删除由你的改动造成的未使用 import、变量或函数。
- 不要删除原本就存在的废弃代码，除非用户要求。

检验标准：每一行改动都应该能直接追溯到用户请求。

### 4. 目标驱动执行

**定义成功标准。循环推进，直到完成验证。**

把任务转化为可验证的目标：

- “添加校验”转换为“为无效输入写测试，然后让测试通过”。
- “修复 bug”转换为“写一个能复现 bug 的测试，然后让测试通过”。
- “重构 X”转换为“确保重构前后测试都通过”。

对于多步骤任务，给出简短计划：

```text
1. [步骤] -> 验证：[检查项]
2. [步骤] -> 验证：[检查项]
3. [步骤] -> 验证：[检查项]
```

强成功标准可以让你独立循环推进。弱成功标准，例如“让它能用”，通常需要不断澄清。

这些准则有效的表现是：diff 中不必要的修改更少，因为过度复杂导致的重写更少，澄清问题发生在实现之前，而不是犯错之后。
